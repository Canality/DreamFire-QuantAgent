## 二、详细修改

### 2.1 多源数据级联获取 (extension.py)

**问题**: 原扩展仅使用 yfinance，在国内网络环境下 (1) 速度极慢 (50s/3只), (2) 经常拉取失败, (3) 失败时无自动降级

**修改**: 实现三层级联数据源:

```
akshare (东方财富) → baostock (自营服务器) → yfinance (Yahoo Finance)
     ↓ 失败              ↓ 补充缺失               ↓ 最后兜底
```

**新增函数**:
- `_data_cache`: 线程安全内存缓存，避免全量价格矩阵通过 LLM 上下文
- `_fetch_baostock()`: 使用 baostock 自有服务器获取数据，无需注册，无频率限制
- `_summarize_top_movers()`: 返回紧凑摘要而非全量数据给 LLM
- `_get_cached_data()` / `_set_cached_data()`: 缓存管理

**修改函数**:
- `_fetch_real_data()`: 从单源 fallback 改为多源级联合并（每层只补充缺失）
- `fetch_data()`: 返回摘要 (n_stocks, date_range, top_movers) 而非原始价格矩阵
- `compute_factors()`: 自动从缓存读取数据，无需 LLM 传递
- 数据源顺序: akshare → baostock → yfinance

**效果对比**:

| 指标 | 修改前 | 修改后 |
|------|--------|--------|
| 数据获取成功率 | 33/49 (67%) | 49/49 (100%) |
| 数据获取耗时 | 328s | 15s |
| 评分影响 | 73.1/100 | 78.5/100 (+5.4) |

### 2.2 移除量化工具 Leader-Only 限制 (tools.py)

**问题**: `_build_quant_tools()` 函数硬编码了 `if role != "leader": return []`，导致 Bull/Bear 子 Agent 无法调用量化分析工具，多 Agent 架构形同虚设

**修改**: 移除角色检查，所有团队成员（Leader + Bull + Bear）均可获得全部 8 个量化工具

**修改前** (`tools.py` line 510):
```python
if getattr(ctx, "role", "") != "leader":
    return []
```

**修改后**: 直接构建工具，无角色限制

**效果对比**:

| 指标 | 修改前 | 修改后 |
|------|--------|--------|
| Coordinator 工具数 | 55 | 55 |
| Bull Analyst 工具数 | 13 (无量化工具) | 21 (含8个量化工具) |
| Bear Analyst 工具数 | 13 (无量化工具) | 21 (含8个量化工具) |
| Bull/Bear 分析质量 | LLM 凭空编造 | 基于真实因子数据 |

### 2.3 自评估框架多源数据支持 (scoring.py)

**问题**: scoring.py 独立实现了数据获取，仅用 yfinance，数据不完整时评分失真

**修改**: 复用扩展的三层级联获取函数，确保评分基于完整数据

### 2.4 缓存机制 (extension.py)

**问题**: 原流程中 quant_fetch_data 返回全量价格矩阵 (49×250 天) 通过 LLM 上下文，超出窗口限制且浪费 token

**修改**: 
- fetch_data 将完整数据存入线程安全内存缓存
- 返回 LLM 的仅为紧凑摘要 (n_stocks, n_days, date_range, top_movers)
- compute_factors 等后续工具自动从缓存读取，无需 LLM 传递数据

**效果**: LLM 上下文从 ~100K chars (全量数据) 降至 ~2K chars (摘要)
