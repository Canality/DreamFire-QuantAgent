### 未使用但在框架中地位很高的

**1. Symphony（技能编排引擎）—— 框架的头牌功能**

Symphony 是 JiuwenSwarm 的**技能关系图谱 + 自动编排系统**。它的工作方式：
- **Score（总谱）**：自动扫描所有已安装的 Skill，分析它们之间的语义关系，构建一张"什么技能可以解决什么问题"的关系图
- **Plan（规划）**：给定用户任务，Symphony 自动规划"应该先调哪个 Skill、再调哪个 Skill、它们之间怎么衔接"——生成一个可视化的 Mermaid 执行图
- **Graph（图谱）**：展示技能之间的依赖关系和协作链路

```
Step 1 → fetch_data
Step 2 → compute_factors
Step 3 → bull_view + bear_view
...
```
这不是 Agent 在做决策——是人在 SKILL.md 里写好了流程。Symphony 的定位恰恰是**取代这种硬编码流程**——让 Agent 自己看图谱、规划步骤。

**2. Skill Evolution（技能自动进化）—— 框架的学习机制**

框架有一个完整的 skill evolution 系统：
- **Team Skill Evolution** (`auto_scan`)：自动扫描技能目录的变化，检测新技能或已有技能的修改，更新技能图谱
- **Completion Followup**：任务完成后，根据执行轨迹自动建议技能改进——"上次这个任务失败是因为缺了 X 工具，要不要加？"
- **Trajectory Registry**：记录每次 Agent 执行的完整轨迹，供后续学习和审计
- **Member Skill Evolution**：单个 Agent 可以基于自己的经验调整行为

我们已经注册了 `swarm.team_skill_evolution` rail（在 multi-agent 日志里能看到它启动了），但 `auto_scan=False`，`completion_followup_enabled=False`——**进化系统被部署了但关了门。**

**3. Cron Tools（定时调度）—— 8 个工具已注册，从未使用**

**4. Sub-Agent 层级（浏览器 Agent / Code Agent）**

Phase A 的原预注册结论必须保留：两因子仍为 `DOES_NOT_QUALIFY`，不能在看到结果后修改阈值并追认晋级。以下发现用于修订**下一轮**研究制度，不回写 Phase A 的历史判定。

#### 两因子实际通过了五项中的四项

| 指标 | 两因子相对六因子 | 原规则判定 |
|---|---:|---|
| 配对中位收益差 | +0.7673pp | 通过 |
| 综合效用胜率 | 15/21（71.4%） | 通过 |
| 最近4窗 | 2/4 | **唯一否决项** |
| 配对中位回撤差 | +0.0998pp | 通过 |
| 最差收益差 | 改善 +1.4123pp | 通过 |

逐窗稳健性复核：

- 15/21 胜率的 Wilson 95% 区间约为 50.0%~86.2%；最近2/4的区间约为15.0%~85.0%，后者几乎没有区分能力。
- 最近5窗为3/5，最近6窗为4/6，最近8窗为6/8；结论会因近期窗口长度的一个小改动而翻转。
- 18组滚动4窗中有14组达到“至少赢3窗”，当前终点恰好落在失败组。
- 三个连续7窗时间块均为5/7获胜，未呈现只依赖单一历史阶段的优势。
- 以长度3做时间区块自助抽样，两因子的配对中位综合效用改善为 +0.3566pp，95%区间约 +0.1156~+1.0222pp；这只能作为开发集不确定性描述，不能冒充未来样本外证明。

因此，两因子更准确的研究状态应为 `CHALLENGER_WITH_RECENT_DECAY`：开发集证据明显优于六因子，但最近两窗分别落后约0.89pp和0.74pp，需要诊断衰减，尚不能无条件切入生产。

#### 下一轮改用分层决策，不再只有通过/淘汰

1. **硬性安全门槛**：无泄漏、49/49与6/6、仓位约束、最差收益和回撤不发生灾难性恶化；这些继续一票否决。
2. **连续证据排序**：使用逐窗 `0.70×收益-0.30×最大回撤` 的配对差、时间区块稳定性和区块自助抽样分布排序候选。
3. **研究状态**：区分 `REJECTED / CHALLENGER / PRODUCTION`。最近4窗降为衰减警报或加权项，不再单独否决。
4. **生产晋级**：必须依赖未来未观察数据或赛事真实评测。若提交前没有新窗口，只能按保守分位综合效用选择 challenger，并明确这是期望效用决策，不是完成了样本外证明。

21个20日非重叠窗口虽然偏少，但属于诚实的有效样本。不能通过加入大量重叠窗口虚增样本量；优先使用更长历史、连续时间块、区块自助抽样和未来窗口。

#### Phase B：有限预算的 2×2 机制实验

当前 `PositionSizer._apply_constraints()` 明确执行 `del scores`：因子得分只决定入选名单，15只股票入选后，权重完全由逆波动率决定。预测强弱没有进入配仓，这是比继续广泛挖因子更明确的改进点。

下一轮只运行以下四个事前固定方案：

| 方案 | 因子权重 | 配仓优先级 |
|---|---|---|
| T0 控制组 | mom20 0.71 / volume_trend 0.29 | 当前逆波动率 |
| T1 收缩组 | mom20 0.85 / volume_trend 0.15 | 当前逆波动率 |
| T2 得分配仓组 | 0.71 / 0.29 | 逆波动率 × 温和得分倾斜 |
| T3 联合组 | 0.85 / 0.15 | 逆波动率 × 温和得分倾斜 |

得分倾斜事前固定为：

```text
priority_i = inverse_vol_i × exp(0.20 × clip(score_z_i, -2, 2))
```

四组都保持正分裸分 Top 15、单股≤10%、板块≤25%、总仓位≤95%，不同时修改选股数量、板块约束和市场状态算法。这个设计分别回答两个问题：`volume_trend` 从29%收缩到15%能否缓解近期衰减；预测得分进入仓位能否把选股优势转换成真实收益。

Phase B 的目标不是承诺固定提分，而是保住两因子约 +0.77pp 的配对收益优势，同时把配对中位回撤恶化控制在 +0.30pp 内，并要求至少2/3个连续7窗时间块综合效用为正。只有完成这轮后，才单独研究基于市场广度/风险的连续总仓位缩放；禁止把因子、配仓和择时一次性同时修改。

### [Codex] 2026-07-21 — 把 Phase B 变成真正的 Agent 量化研究闭环

#### 当前状态评估

Goone 指出的方向问题成立：当前系统已经证明了“多个 Agent 确实运行并分别调用 RPC”，但 Bull/Bear 仍主要是在同一份固定结果上生成两种文字视角；策略提出、实验设计、证据反驳、历史经验检索和晋级决策仍由人手工完成。它是**真实多 Agent 工具链**，还不是一个充分发挥 Agent 推理能力的量化研究系统。

Missed 完成的 Phase B 2×2 实验提供了最适合改造的真实案例：

| 方案 | 配对中位收益差 | 配对中位回撤差 | 效用胜率 | 7窗正效用块 |
|---|---:|---:|---:|---:|
| T0 两因子控制 | +0.7673pp | +0.0998pp | 15/21 | 3/3 |
| T1 收缩 volume_trend | +0.0000pp | +0.1363pp | 10/21 | 1/3 |
| T2 得分进入配仓 | **+0.9102pp** | +0.0986pp | 15/21 | 3/3 |
| T3 收缩+得分配仓 | +0.3216pp | +0.0262pp | 13/21 | 2/3 |

结果说明：`volume_trend` 从0.29收缩到0.15的假设没有得到支持；真正有效的机制是让预测得分温和进入仓位。T2 是当前最强 challenger，但所有21窗仍属于已观察开发数据，不能宣称已经样本外晋级。

#### 对 Goone 框架建议的代码核验

Goone 提到的 Symphony、Evolution、Experience Bank、Cron 在仓库中确实存在，但当前成熟度必须诚实描述：

1. Symphony 已有 `compose/read/refresh score` 等工具，能够返回 Skill 执行计划和 Mermaid 图；**尚未证明生成的图能正确执行完整量化任务**，不能把“生成规划”写成“已替代硬编码工作流”。
2. Experience Bank 有采集、蒸馏、检索实现；**尚未接入量化实验结构化结果**。
3. Team/Member Skill Evolution 已装配轨迹注册器，但 `auto_scan=false`，团队 registry 当前是进程内存级；不能写成已经跨任务持续学习。
4. Cron 工具已构建，但对当前初赛收益没有直接贡献，应放在核心研究闭环之后。

还发现一个 P0 风险：现有量化 Skill 文档仍包含旧事实，例如 `data-fetcher` 写 yfinance→akshare、失败降级模拟数据和前复权；`stock-selector` 写每板块至少一只；`report-generator` 写收盘成交和前复权；`quant-investment` 仍引用旧开发/封存窗口。当前真实实现已经是五源逐只补缺、失败关闭、原始行情、裸分Top 15和首日开盘固定股数。**如果现在直接让 Symphony 索引这些 Skill，它会自动编排出一条过期且错误的流程。**

#### 下一步假设

> Agent 的主要价值不是自由搜索更多参数，而是自主完成“检索历史证据→提出有限可证伪假设→Bull主张→Bear反驳→事前登记→调用确定性实验器→审计晋级→沉淀经验”的研究闭环；数值计算和约束执行继续由可复现代码负责。

#### 目标架构：Agent Research Council

```text
用户目标：提高收益/回撤综合效用
  ↓
Symphony：根据当前任务规划需要的 Skills，并输出可审计执行图
  ↓
Experience Retriever：检索相似市态、历史候选及失败原因
  ↓
Bull / Alpha Advocate：只提出一个机制假设和有限候选
  ↔
Bear / Risk Falsifier：寻找泄漏、过拟合、尾部风险和反例
  ↓
Coordinator：冻结 preregistration、候选预算和验收协议
  ↓
Deterministic Experiment Runner：运行统一快照与逐窗评测
  ↓
Evidence Auditor：核对覆盖率、因果性、约束、配对统计与产物哈希
  ↓
Coordinator：输出 REJECTED / CHALLENGER / PRODUCTION 状态
  ↓
Experience Bank：保存结构化证据；Evolution 只提出流程改进建议
```

这里 Agent 负责开放世界中的判断、规划、争论和复盘；Extension/评测器负责精确计算。不能让 LLM 直接计算收益率、修改回测结果或绕过硬性护栏。

#### 分阶段实践方向

**P0：先修 Skill 事实契约，禁止错误知识进入 Symphony**

- 把六个 quant-finance Skill 的数据源、复权口径、选股规则、成交口径、策略状态和 fail-closed 条件同步到当前代码。
- 为每个 Skill 写清机器可检查的输入、输出、前置条件和失败条件；特别是 `fetch→factor→select→allocate→backtest` 的缓存依赖。
- 增加“Skill 声明与生产配置一致性”测试；README/Skill/Extension 任一处声称的能力必须有对应验收证据。

验收：旧三源、模拟数据、前复权、板块保底、旧封存集等陈述不再被 Symphony 索引；文档一致性测试失败时禁止正式运行。

**P1：做一个受约束的 Symphony 量化规划 POC**

- 先只覆盖三个任务：正常投资分析、数据覆盖失败恢复、候选策略衰减诊断。
- Symphony 负责选择和排序 Skills；生成的计划先经过 Policy Validator，强制检查数据获取必须先于因子计算、覆盖不足必须停止、回测必须使用已冻结策略 ID。
- Coordinator 按验证后的图执行，记录计划图、实际 RPC 轨迹和偏差；不能只展示一张 Mermaid 图就算成功。

验收：三类任务各运行一次；计划步骤、实际工具归属、缓存依赖和最终产物全部可对齐，故意制造49只覆盖不足时能失败关闭。

**P2：把 Phase B 变成第一条 Agent 自主研究轨迹**

- 当前 T2 固定为 challenger 基线：因子仍为 mom20 0.71 + volume_trend 0.29，配仓为 `inverse_vol × exp(0.20×clip(score_z,-2,2))`。
- Bull 读取 Phase A/B 结构化结果，只允许提出一个下一步机制假设；Bear 必须给出反例窗口、可能恶化的风险指标和否决条件。
- Coordinator 只能从事前注册的离散 StrategySpec 中选候选，不能接受 LLM 临场生成任意小数权重。
- 实验前先输出 preregistration JSON，实验后由独立 Auditor 读取产物判定，Coordinator 不得自行改阈值。

建议下一项只研究**连续风险暴露**，不再改因子：在市场风险升高时将总仓位从95%平滑降低，目标是在尽量保留 T2 收益优势的同时改善尾部回撤。具体公式和最多2个候选必须在运行前冻结。

**P3：接入结构化 Experience Bank，而不是保存散文总结**

每条经验至少包含：`snapshot_hash`、Git commit、StrategySpec ID、决策日与窗口、regime、候选假设、Bull主张、Bear反例、配对收益/回撤/效用、三个时间块结果、验收状态、失败原因和证据路径。

下一次 Agent 提案前必须检索相似 regime 与相同机制的历史记录，并在 preregistration 中引用经验 ID。检索结果只能帮助提出假设，不能把测试窗口答案注入当期决策。

验收：在新会话中询问“得分倾斜是否有效”，Agent 能检索到 T2 证据和 T1 失败原因，并准确指出这些只是开发集证据。

**P4：Evolution 采用 suggestion-only，禁止自动改策略**

- 暂不打开 `auto_save`，也不允许 Evolution 根据一次回测自动修改因子权重。
- completion followup 只可提出 Skill/流程修订，例如发现遗漏覆盖检查、角色未真实委派、报告引用错误口径。
- 每个建议必须附轨迹证据，通过人工批准、单测和端到端验收后才能写回 Skill。

这能展示“系统从失败中学习”，同时避免自进化退化成自动过拟合。

**P5：最后再做 Cron/主动监控**

核心闭环通过后，再用 Cron 每日检查数据源健康、市场状态变化和组合风险；变化只触发重新分析和提醒，不自动改权重或交易。Browser新闻、MCP和AutoHarness暂缓：它们工程面更大，而且新闻回测容易引入发布时间泄漏，当前不能为了框架展示牺牲测量可信度。

#### 预期效果

- **策略侧**：T2 当前开发集相对六因子的配对中位收益优势为 +0.91pp、配对中位回撤恶化约 +0.10pp；这是下一轮基线，不是未来收益承诺。风险暴露实验的目标是减少坏窗口损失，而非继续放大历史中位收益。
- **Agent侧**：从“固定RPC流水线+文字Bull/Bear”升级为“规划、检索、对抗论证、预注册、实验、审计、学习”全链路，每项能力都有轨迹和失败测试，而不是按框架功能数量计分。
- **答辩侧**：可以展示同一个 Phase B 案例如何由 Agent 提出、被反方挑战、由确定性工具验证并沉淀为下一轮可检索经验；这比单纯展示 Symphony/Evolution 开关更有说服力。

#### 风险点

1. **框架剧场**：只生成规划图或打开开关，实际决策仍是硬编码；必须对齐“计划图—实际轨迹—结果证据”。
2. **自主过拟合**：Agent 不受限地生成权重会迅速榨干21窗；必须限制候选预算和离散配置。
3. **错误知识放大**：Skill 文档过期时，Symphony 会更稳定地执行错误流程；所以 P0 先于所有框架接入。
4. **经验泄漏**：Experience Bank 若混入前向答案，会形成隐蔽前视；经验必须标注决策时可见范围。
5. **自动进化污染**：一次失败轨迹不足以修改策略；Evolution 先只改流程建议，不改投资参数。

#### 教学要点

- Agent 的强项是处理不完全信息下的规划、辩论和经验迁移；精确收益计算应交给确定性程序。
- “拥有记忆”不等于“拥有证据”，Experience Bank 必须保存口径、时间边界和产物哈希。
- Symphony 是 planner，不天然等于 executor；真正的 Agent 闭环必须证明计划被正确执行。
- Bull/Bear 的价值不在生成两段相反文案，而在形成可证伪的主张与反例，并影响是否允许运行或晋级。
- 自进化最先应该学习流程错误和能力缺口，而不是从同一批历史窗口持续调参。

### [Codex] 2026-07-21 — Agent 能力如何传导到初赛收益与回撤

#### 核心判断

Agent 能力本身不会自动提高初赛成绩。只有当它在评测决策时**改变最终股票、个股权重或现金比例**，并且这种改变通过因果统一评测证明优于固定策略，才形成初赛 alpha。没有改变组合的 Symphony 图、Cron 调度或 Evolution 建议，主要提升工程可靠性和决赛答辩，不应包装成收益提升。

| 能力 | 是否改变组合 | 初赛直接价值 |
|---|---|---|
| T2 得分进入配仓 | 直接改变15只股票内部权重 | 高，已有开发集正面证据 |
| Agent 风险暴露选择 | 改变总仓位/现金 | 中高，目标是改善坏窗口和回撤 |
| 官方公告风险覆盖 | 排除或降权重大风险股票 | 待验证，可能有效也可能错过反弹 |
| Experience Bank | 帮助选择已注册策略/仓位档 | 间接，必须实际改变决策才有分数影响 |
| Bull/Bear 对抗 | 可能否决高风险动作 | 间接，价值取决于否决是否提升样本外效用 |
| Symphony | 选择和排序分析 Skills | 不直接产生收益，降低漏步骤和错误调用风险 |
| Cron | 定时触发数据/风险检查 | 固定持有20日口径下几乎不直接影响分数 |
| Evolution | 改进后续 Skill/流程 | 本次初赛直接价值低，长期和答辩价值高 |
| 新闻情绪自由选股 | 会改变持仓 | 高不确定性，未经 point-in-time 验证可能降分 |

#### 当前最明确的直接提分机制

Phase B 的 T2 不是“Agent 写了一份更好的报告”，而是将预测强度真正传递到仓位：

```text
T0：Top 15 → 只按逆波动率配仓
T2：Top 15 → 逆波动率 × 温和得分倾斜
```

T2 相对生产六因子的开发集证据为：配对中位收益差 +0.9102pp、配对中位回撤差 +0.0986pp、综合效用胜15/21、三个7窗时间块3/3为正。它是当前最值得接入生产双路径并做未来验证的 challenger；以上数字不是未来窗口收益承诺。

#### 面向初赛的受限 Agent 决策器

不让 LLM 自由生成股票和任意小数权重，而是在 T2 确定性基础组合之上，只允许选择少量预注册动作：

```text
T2 生成基础 Top 15 与内部权重
  ↓
Experience Bank 按 as_of_time 检索相似市态和历史机制结果
  ↓
Bull 论证收益机会；Bear 检查尾部风险与官方重大公告
  ↓
Coordinator 只能选择：
  A. T2 + 95%总仓位
  B. T2 + 85%总仓位
  C. T2 + 75%总仓位
  D. 对命中已注册官方风险事件的股票执行固定降权/替换规则
  ↓
确定性 PositionSizer 重新施加单股10%、板块25%和现金约束
```

这些档位只是动作空间示意，不能直接投入实验；风险指标、映射公式和最多两个候选需在运行前冻结。Agent 的价值在于根据当时可见证据选动作，最终权重仍由代码生成。

#### Experience Bank 对分数的真实作用

Experience Bank 只有在以下链路成立时才影响初赛：

```text
相似历史证据 → 选择不同的已注册策略/仓位档 → 最终组合变化 → 未来收益/回撤变化
```

例如检索到“高波动、动量扩散下降”的历史环境中满仓 T2 尾部较差，Agent 才可能选择85%仓位。必须用逐窗经验回放验证：每个历史决策日只能检索此前已经形成的经验。若只是把 Phase B 总结放进最终报告，分数影响为零。

#### Cron 与公告数据的分数边界

- 当前比赛代理是首日开盘买入、固定股数持有20日。如果赛事不允许持有期再平衡，Cron 每日监控不能擅自换仓，因此主要防止数据过期、覆盖不足和提交错误，不直接提高收益。
- 官方公告可以在决策日前影响选股，但第一版应做风险覆盖而不是自由情绪交易：重大处罚、立案、诉讼、业绩大幅预减、减持或退市风险经 Bear 核验后，触发固定降权/替换候选。
- “公告真实”不等于“公告有 alpha”。负面消息可能已经被价格消化，也可能随后反弹；必须比较 `T2` 与 `T2+公告风险覆盖` 的 point-in-time walk-forward 结果。没有可靠的历史首次发布时间和版本归档前，不得声称能提分。

#### 面向初赛的实施优先级

1. **T2 生产化**：统一 StrategySpec、研发直跑和 JiuwenSwarm 正式路径，完成双路径验收。
2. **连续风险暴露实验**：只研究总仓位映射，目标是减少 T2 的最差窗口损失，不同时改因子。
3. **受限 Agent 策略选择器**：在预注册的 T2 仓位方案中选择，做严格的历史经验回放。
4. **官方公告风险覆盖实验**：先建立 point-in-time 数据库，再决定是否进入选股。
5. **Experience Bank**：为第3、4项提供时间安全的证据检索，而不是充当调参记忆。
6. **Symphony / Evolution / Cron**：服务于规划、审计、持续运行和答辩，不能挤占前四项的提分验证预算。

最终目标不是“用了多少 Agent 功能”，而是证明同一决策时点下：受限 Agent 组合的配对综合效用稳定优于固定 T2。如果做不到这一点，Agent 层应退回解释、审计和监控角色，不应为了展示自主性干预持仓。

## PA_Agent 的核心设计：一个我们能学但没学的东西

PA_Agent 不是一个量化系统——它不做回测、没有因子、不跑 IC。它做的事情是：

> **让 AI 像交易员一样思考：先诊断市场处于什么状态，再决定用什么策略应对。**

我们的系统正好相反：因子算完→排序→选股，**跳过了"诊断市场状态"→"选择策略"这个决策环节。** 当前 regime 系统只是对同一批因子调权重，不是你真正想要的"判市决定策略"。

## 可复用 1：分析检查单 → 改造 Bull/Bear SKILL.md

PA_Agent 有一个 `逐棒分析检查单.txt`——不是泛泛的"你分析一下"，而是逐条 checklist：

```
□ 当前 K 线是阳线还是阴线？
□ 相对于前一根 K 线，收盘价是更高还是更低？
□ 是否突破了关键支撑/阻力位？
□ ...
```

我们的 Bull Analyst SKILL.md 现在是：

```markdown
你的任务是使用 momentum_20、momentum_60、volume_corr 三个趋势因子
独立计算趋势得分，选出 Top 15 趋势最强股票。
```

这和"逐棒分析检查单"的差距是：PA_Agent 给了 AI 一个**不可跳过的思考流程**，我们给了 AI 一个**任务描述**。

**具体改法**：给 Bull 和 Bear 各写一份分析检查单，格式参照 PA_Agent：

```markdown
# Bull Analyst 分析检查单

## 第一步：市场环境扫描
□ 当前判市是什么？（bull / bear / range）
□ 双信号是否一致？（共识 = 高信心，分歧 = 低信心）
□ 波动率处于历史什么水平？（当前 vol vs 半年中位数）

## 第二步：因子信号交叉验证
□ momentum_20 和 momentum_60 方向是否一致？
  → 一致 = 趋势明确，不一致 = 趋势可能转折
□ volume_trend 是正在放大还是萎缩？
  → 量增 = 资金关注度上升，量缩 = 动能衰减

## 第三步：板块轮动扫描
□ 过去 5 个窗口，动量最强的板块是否有持续性？
□ 是否有板块从底部首次进入 top 3？

## 第四步：选股与风险标注
□ 输出 Top 15 + 每只股票的选择理由（不超过一句话）
□ 标注"趋势不稳定但分数高"的股票（mom_20 和 mom_60 方向不一致的）
```

Bear 同理——只是检查的问题不同（尾部风险、量价背离、回撤集中度）。

**工作量**：改两个 SKILL.md，不碰 Python。但效果是 Bull/Bear 的思考过程从"一段话"变成"一份清单"。

## 可复用 2："识别→策略"解耦 → 改造判市系统

PA_Agent 最精妙的设计：**每个市场场景拆成两个文件——"分析识别"和"交易策略"。**

```
上涨通道分析识别.txt    ← 回答"现在是上涨通道吗？有哪些证据？"
上涨通道交易策略.txt    ← 回答"上涨通道里应该怎么做？"
```

识别逻辑稳定（通道的定义不随风险偏好改变），策略可以按风险偏好换（激进策略 vs 保守策略用同一份识别结果）。

**我们当前的 regime 系统没有这个解耦**——`get_regime_weights()` 把"判市"和"因子权重"搅在一起：

```python
if regime == "bull":
    momentum_20 *= 1.5
    reversal_5  *= 0.3
elif regime == "bear":
    momentum_20 *= 0.3
    reversal_5  *= 1.5
```

这是**硬编码的映射表**，不是"识别→策略选择"。

**具体改法**：

```python
# 解耦为两步
# Step 1: 识别（不改，已有 RegimeFusion.detect_with_detail）
regime_detail = RegimeFusion.detect_with_detail(history, index_prices)
# → {"final": "bull", "consensus": True, "vol_ratio": 1.2, ...}

# Step 2: 策略路由（新增——基于识别结果，选择预注册的策略配置）
strategy = StrategyRouter.select(
    regime=regime_detail["final"],
    confidence=regime_detail["consensus"],
    vol_ratio=regime_detail["vol_ratio"],
)
# → StrategySpec(id="bull_high_confidence", factor_weights={...}, position_caps={...})
```

`StrategyRouter` 是一个独立的决策模块，不混在因子计算里。好处：
- 判市逻辑稳定，策略配置可以独立迭代
- 决赛报告里可以展示"因为判市 bull + 高信心 → 选择了激进策略"
- 新增策略配置不需要改因子代码

**和 PA_Agent 的对应**：
- `RegimeFusion.detect()` = "分析识别"
- `StrategyRouter.select()` = "交易策略选择"
- `StrategySpec` = 预注册的策略文件（类似 PA_Agent 的 `上涨通道交易策略.txt`）

## 可复用 3：经验库按市态分类 → 改造我们的回测结果存储

PA_Agent 的 `experience/` 目录：

```
experience/
├── trending_tr/      ← 趋势交易区间下的历史案例
```

每次 AI 分析时，会检索**同类型场景**的历史案例作为参考。

我们有 21 个窗口的完整回测数据，但它们是按时间排列的——不是按市态索引的。

**具体改法**：在 Codex 设计的 Experience Bank 里，把每个窗口的结构化经验按 regime 分类存储：

```json
{

  "strategy": "T2_score_tilted",
  "return": +3.42,
  "max_dd": 1.87,
  "top_sector": "科技/AI/半导体",

}
```

下一次 Agent 在"bull + 高信心"环境下做决策时，Experience Retriever 可以检索所有标签匹配的历史窗口，输出"历史上类似的市态下，T2 策略中位收益 +X%，最差 -Y%"。**这比"21 窗口中位 +0.38%"的汇总数字有用得多。** 这也是 PA_Agent 经验库给我们的核心启发——经验的价值在于**可检索的上下文**，不是统计摘要。

## 可复用 4：决策树 → 让 Coordinator 的综合流程可视化

PA_Agent 有一个"二元决策树"——每一步都是闸门（gate），只做二选一：

```
Gate 1: 市场有明确方向吗？
  → 有 → Gate 2
  → 没有 → 不做（或震荡策略）

Gate 2: 方向有持续性证据吗？
  → 有 → Gate 3
  → 没有 → 减仓
...
```

**具体改法**：Coordinator 的综合 Bull/Bear 结果可以做成同样的决策链：

```
Gate 1: Bull 和 Bear 的推荐 overlap > 30% 吗？
  → 是 → Gate 2（高共识路径）
  → 否 → 分歧处理路径

Gate 2: 判市信心是高吗？
  → 是 → 满仓（95%）+ Bull 推荐为主
  → 否 → 降仓（85%）+ 风控优先

Gate 3: 是否有 Bear 明确警告的股票？
  → 是 → 剔除或降权
  → 否 → 采用 Bull 推荐权重
```

每一步都是二元分支，最终输出一个可解释的决策路径。决赛答辩时展示"Coordinator 在这个窗口走了 Gate1→Gate2→...→最终组合"，比"综合 Bull 和 Bear 的意见后选择了 15 只股票"有说服力得多。

## 不应该照搬的东西

PA_Agent 有几样东西**不适合**我们：

1. **不跑回测**：PA_Agent 决策后不验证——它的"验证"是下一根 K 线。我们是量化系统，回测是核心。不学这个。
2. **自由文本分析**：PA_Agent 的输出是自然语言（"建议做多，止损设在 1.2350"）。我们的输出必须是结构化持仓（`{"000333.SZ": 0.08}`）。PA_Agent 让 LLM 自由发挥的风格不适合我们的核心链路。
3. **无约束检查**：PA_Agent 没有 fail-closed。AI 说买什么就展示什么。我们必须保留硬性安全门槛。

#### 对四项提议的逐项结论

| 提议 | 结论 | 原因与安全落地方式 |
|---|---|---|
| Bull/Bear 长检查单 | **有条件同意，当前版本不能直接用** | 当前工具并未稳定提供“半年度波动率基线、过去多个窗口板块持续性、首次进入 top3、单日暴跌分布”等全部字段。检查单只能询问 RPC 明确返回的可观测字段；缺失必须输出 `unknown`，不能让 LLM 推测。输出应是经 schema 校验的 `evidence[] / warnings[] / candidates[]`，而不是自由文本理由。 |

| Experience Bank 按市态索引 | **同意，但 regime 不能是唯一索引** | 每条经验至少要保存 `as_of_time`、`available_at`、数据快照 hash、决策窗、评估窗、strategy spec id、`research_only/decision_safe`。检索前先按时间可用性过滤，再按 regime 等上下文排序；bear 样本为 0 时必须明确“无历史证据”，不能生成统计结论。 |
| Coordinator 决策树 | **同意做解释层，不同意当前阈值** | `overlap > 30% → 95% 仓位`、低信心→85% 等规则没有样本外证据，不能直接进入生产。先可视化已有硬门：数据覆盖、策略 ID、15 只/6 板块、单股/板块 cap、现金下限、证据时点和角色所有权。新的风险动作必须经 Phase C 验证。 |

#### 两个必须先纠正的问题

**1. 当前 quant-investment Skill 已经出现“提示词能力超前于工具能力”。** 其中包含多个工具没有提供或没有可靠时点定义的检查项，还保留了“21 个 Walk-Forward IC 窗口”等旧口径；+0.0787/72.7% 与 +0.0315/81.8% 来自原先的 11 个开发 IC 窗口，不应改写成 21 窗口。此外，Bull/Bear 自由建议 70%–95%/30%–70% 总仓位、Coordinator 自由调仓，与当前冻结动作空间冲突。P0 不能只改三个 Skill 后就声称“6 个 Skill 已全部修复”，必须逐文件建立事实清单并以代码/产物为证据。

**2. 当前 `evaluation/symphony_poc.py` 不是 Symphony POC，而是 PolicyValidator 原型。** 它没有调用 Symphony 生成计划，验证的是代码内手写的 `TASK_SPECS`；放宽 R1 后 3/3 通过，只证明校验器接受这三份静态样例，不能证明 Symphony 会正确规划或实际执行。这属于当前最典型的“伪集成”：命名和提交说明声称接入框架，运行链路却没有经过框架。

真正的 POC 至少需要留下以下证据链：

```text
真实任务输入
  → Symphony 实际生成的 plan artifact
  → plan 规范化 + PolicyValidator
  → 实际 Coordinator/graph 执行
  → RPC trace 与 plan 对账
  → 故意缺步骤/坏数据时 fail-closed
```

在做到以前，应把它改称 `PolicyValidator prototype`，不要作为“Symphony 已接入”的完成证据。

#### 建议的下一步顺序

**Phase Agent-A（只改架构与可解释性，不影响评分动作）**

1. 定义真实的 `RegimeDiagnosis` schema，只包含 `detect_with_detail()` 当前确实返回的 `final / technical / index / consensus`；如果需要 `vol_ratio`，先实现确定性计算、时点定义和单测，不能在示例里假设它已经存在。
2. 新增纯函数 `AnalysisPlaybookRouter`，只选择 Bull/Bear 的分析模板，不改变 T2 因子、股票排序或仓位。
3. Bull/Bear 输出改为结构化 schema；每条判断必须引用输入字段，字段缺失即 `unknown`。
4. 增加性质测试：相同输入路由结果相同、只返回注册 playbook、未知状态不触发策略动作、缺数据不调用 LLM。
5. 让真正的 Symphony 生成计划并保存 plan/trace，对照验证实际执行；否则暂不宣称框架集成完成。

**Phase Agent-B（可能影响初赛分数，必须独立预注册）**

在 Agent-A 完成后，只允许测试最多两个简单、可证伪的 T2 风险敞口映射；使用时间因果的开发窗重放，并报告配对收益、回撤、最差窗与按 regime 的样本数。当前 bear=0，因此禁止据此上线 bear 专用策略。只有通过预注册门槛，StrategyRouter 才能从“解释路由”升级为“交易动作路由”。

**结论：** Goone 提出的”识别和策略分层”是对的，但”先写长检查单、再立刻让路由选多套仓位/权重”顺序不对。先把 Agent 变成一个**输入可追溯、输出可校验、路由可复现、失败会关闭**的系统，再给它新增决策自由度。这既更符合比赛的 Agent 核心，也能避免再次出现 README/Skill 声称能力已经存在、真实代码却没有执行的情况。
