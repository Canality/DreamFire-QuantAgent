# 2026-07-21 双路径真实运行验证

## 验证目标

验证以下两条现有执行路径是否能够在当前环境端到端运行：

1. `scripts/run_quant_pipeline.py`：绕过 Agent 的策略研发旁路。
2. `evaluation/run_multi_agent.py`：JiuwenSwarm Coordinator + Bull + Bear 正式路径。

本轮不修改策略代码，不作为新策略版本或评分实验。

## 环境

- 项目虚拟环境：Python 3.11.2
- pandas 3.0.2、numpy 2.4.4
- akshare 1.18.64、baostock、yfinance 1.5.1
- LLM：DeepSeek `deepseek-chat`

## 路径一：研发旁路

### 运行结果

- 运行完成并生成 `output/pipeline_results.json`。
- yfinance 49 只全部受限流影响；akshare 仅取得 14/49 只。
- 数据只有金融、消费两个板块，共 118 个交易日。
- 判市：range。
- 选股步骤报告 9 只、2 个板块，但配仓步骤实际持有 14 只。
- 总仓位 94.99%，金融 47.50%，消费 47.49%。
- 累计收益 -1.24%，年化收益 -2.65%，最大回撤 13.72%，Sharpe -0.1621，年化波动率 16.36%。

### 判定

**未通过有效策略验证。** 脚本技术上跑完，但存在以下问题：

1. 数据覆盖率只有 28.6%，没有按 README 所述逐层补齐全部股票。
2. `select_stocks()` 的结果没有传入 `PositionSizer.allocate()`；配仓直接使用完整 scores 的 Top N，导致选股和持仓不一致。
3. 两个板块各约 47.5%，违反单板块不超过 25% 的声明。约束处理后再次归一化，把已经压低的板块权重重新放大。
4. 使用期末因子选择股票后，在同一整段历史价格上回测，存在前视偏差，指标不能作为样本外策略表现。

## 路径二：JiuwenSwarm 多 Agent

### 真实执行证据

- 成功加载 `quant_team`，包含 leader、bull_analyst、bear_analyst。
- 三个 Agent 均实际调用 DeepSeek。
- Leader 创建了包含 fetch、factor、Bull、Bear、选股配仓、回测报告的任务 DAG。
- Bull 和 Bear 实例收到团队消息并进入运行状态。
- Leader 实际调用模型工具 `quant_fetch_data`。

### 失败表现

- `quant_fetch_data` 连续调用 34 次，始终没有进入 `quant_compute_factors`。
- 为避免无意义消耗 LLM token，人工终止测试进程。
- 运行日志没有 QuantFinance Extension handler 的加载或注册记录。
- `evaluation/run_multi_agent.py` 及其团队装配调用链中没有初始化 `ExtensionRegistry`、加载 `ExtensionManager` 的代码。

### 隔离冒烟测试

手动实例化 `QuantFinanceExtension` 并注册到测试 registry 后：

- 8 个 RPC handler 全部注册成功。
- `quant.compute_factors` 在合成数据上成功。
- `quant.bull_view` 在合成数据上成功。
- `quant.bear_view` 在合成数据上成功。

因此 handler 本身可运行；失败发生在正式入口没有把 Extension 注册表和 Agent 工具层接通。

### 其他问题

- embedding provider 未配置，团队记忆工具初始化失败。
- Windows GBK 日志无法编码 emoji，产生 logging error，但不是本次主流程失败原因。
- 当前验证脚本以“存在文本且存在工具调用”作为多 Agent 成功条件，判据过宽；即使量化 RPC 全部失败也可能被误判为成功。

## 总结

- 研发旁路：**能结束，但结果无效，不应视为回测通过。**
- 多 Agent 路径：**多 Agent 编排真实存在，但端到端量化流程未跑通。**
- Claude 确实实现并测试过若干独立组件和离线实验，但当前代码不能证明做过一次成功的、覆盖真实 49 只股票的正式端到端验收。
