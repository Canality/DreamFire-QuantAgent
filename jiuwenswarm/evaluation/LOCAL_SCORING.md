# 本地代理评分机制（local_proxy_score_v1）

## 定位

本评分器用于官方横截面分布和资源基准公布前的本地比较，不是官方分数。它只读取统一评测器已经生成的逐窗结果，不自行抓行情或维护另一套回测逻辑。

官方已明确：投资组合80分，其中收益占70%、最大回撤占30%；资源20分，其中Token/运行/算力分别为10/5/5。官方尚未明确收益与回撤采用何种参赛队伍标准化，也未公布Token和运行时基准。

官方文本存在内部冲突：Token标题为10分、运行标题为5分，但规则正文分别写达标得15分和10分。由于资源维度总计20分，本地代理按声明的10/5/5封顶，并在结果JSON中保留该假设。

## 投资组合80分

冻结生产六因子21个因果、非重叠、20交易日窗口作为本地参考分布。对候选的每个同日期窗口：

```text
收益分 = 56 × 收益在参考分布中的经验百分位
回撤分 = 24 × (回撤越小越高的经验百分位)
窗口投资分 = 收益分 + 回撤分
```

并列值使用中点秩。主报告使用21窗投资分的均值，另外报告中位数、P10和最差值。生产策略相对自身参考的期望投资分按构造为40/80；候选高于40表示平均历史排名优于生产参考，不代表官方会得到相同分数。

选择这种机制的原因：官方允许标准化排序，但参赛队伍分布未知。固定经验分布保持单调、可复现，并避免继续使用旧脚本中人为设置的收益率分段。

## 资源20分

- Token（10分）：实际用量不超过基准得10分；每完整超出10%扣2分。
- 运行效率（5分）：实际时长不超过基准得5分；每完整超出10%扣1分。
- 算力经济性（5分）：按官方档位优/良/中/差映射5/3/1/0。

缺少真实用量或官方基准时，该项为 `pending`。只要有任何资源项 pending，就不输出精确100分总分，而输出当前可确定的总分区间。

不得使用估算Token、假定内存或擅自选择算力档位填充正式结果。

## 使用方式

Phase B 结果必须包含 `details` 逐窗证据：

```powershell
.venv\Scripts\python.exe evaluation\phase_b_experiment.py `
  --snapshot evaluation\data_snapshots\sina_20260721_135352
```

资源基准未知时：

```powershell
.venv\Scripts\python.exe evaluation\local_scoring.py `
  --results evaluation\phase_b_latest.json `
  --strategy phase_b_t2_score_alloc `
  --reference-strategy production_six_factor `
  --output evaluation\local_score_phase_b_t2_latest.json
```

官方基准公布并取得真实测量后，可追加：

```text
--actual-tokens ... --baseline-tokens ...
--actual-runtime-seconds ... --baseline-runtime-seconds ...
--compute-tier excellent|good|medium|poor
```

T2 的本地投资代理分比生产高3.57分，但P10仍低，说明平均历史排名改善而尾部风险仍是下一阶段重点。资源三项均 pending，因此T2当前只可报告总分区间43.57~63.57/100。

## 禁止解释

- 不得把43.57或63.57写成官方成绩。
- 不得跨不同快照、窗口或成交口径比较分数。
- 不得把开发窗口的代理分当成未来样本外证明。
- 官方发布横截面公式或资源基准后，应新增方法版本，不静默改写v1历史结果。
